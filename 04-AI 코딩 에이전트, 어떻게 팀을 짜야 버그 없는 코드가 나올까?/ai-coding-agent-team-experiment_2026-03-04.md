# AI 코딩 에이전트, 어떻게 팀을 짜야 버그 없는 코드가 나올까?

> 최종 수정: 2026-03-04

## TL;DR

AI에게 코드를 "한 번에 잘 짜라"고 하면 Critical 버그가 남는다.
"짜고 → 검증하고 → 고쳐라"는 루프를 돌리면 버그가 0에 수렴한다.
사람의 코드 리뷰와 같은 원리다.

## 1. 실험을 왜 했나?

Claude Code로 AI 에이전트 팀을 꾸려 코드를 짤 수 있다.
문제는 **"어떻게 팀을 구성하느냐"에 따라 결과물 품질이 크게 달라진다**는 것.

어떤 구조가 가장 좋은지 감이 아니라 데이터로 확인하고 싶었다.

## 2. 실험 설계

### 과제

할 일 관리 앱 (Todo App) 하나를 처음부터 끝까지 만들기.
기능: 추가, 삭제, 완료 체크, 필터, 개수 표시, 로컬 저장, 에러 처리, 접근성

### 4가지 팀 구성

AI 에이전트 팀을 4가지 방식으로 구성했다.

| | 한 번에 끝 (1회성) | 검증 후 수정 (반복) |
|--|---|---|
| **같은 공간에서 작업** | A. Agent Team | C. Ralph Loop + Team |
| **각자 격리해서 작업** | B. Agent Team + Worktree | D. Ralph Loop Parallel |

> **1회성**: 개발자가 코드를 짜면, 리뷰어가 검토하고 끝. 수정 기회 없음.
> **반복**: 개발자가 코드를 짜면, 리뷰어가 검토하고, 버그가 있으면 수정 후 재검토. 버그 0이 될 때까지 반복.

### 통제한 변수

모든 실험에서 아래 조건은 동일하게 유지했다.

| 항목 | 내용 |
|------|------|
| AI 모델 | Claude Sonnet 4.6 (개발, 리뷰) / Claude Opus 4.6 (팀 리드) |
| 도구 | Claude Code v2.1.63 |
| 기획서 | PM 에이전트가 작성한 통합 스펙 (PRD + 유저플로우 + 에러 시나리오) |
| 에이전트 | 개발자(7개 스킬), 리뷰어(6개 스킬), 테스터(6개 스킬) |
| 프롬프트 | 동일한 주의사항 명시 |
| 프레임워크 | Next.js 16 + TypeScript + TailwindCSS |

**바꾼 것은 "팀 구성 방식" 하나뿐.**

## 3. 결과

### 핵심 수치

| 팀 구성 | 평균 소요 시간 | Critical 버그 0 달성 |
|---------|--------------|---------------------|
| A. Agent Team (1회성) | ~8분 | **0/2** (0%) |
| B. Team + Worktree (1회성) | ~8분 | **0/2** (0%) |
| C. Ralph Loop + Team (반복) | ~10분 | **2/2** (100%) |
| D. Ralph Loop Parallel (반복) | ~14분 | **2/2** (100%) |

### 한눈에 보기

```
1회성 조합 (A, B)  →  Critical 0 달성률:   0%  (0/4)
반복 조합 (C, D)   →  Critical 0 달성률: 100%  (4/4)

8번의 실험에서 단 한 번도 예외 없음.
```

## 4. 왜 이런 결과가 나올까?

### 1회성의 한계

```
개발 → 리뷰 → "버그 3개 발견" → 끝 (수정 기회 없음)
```

리뷰어가 아무리 정확하게 버그를 찾아도, **수정할 단계가 없다.**
보고서만 남고 코드는 그대로다.

### 반복의 강점

```
개발 → 리뷰 → "버그 3개" → 수정 → 재리뷰 → "버그 0개" ✅
```

리뷰에서 발견된 버그를 수정하고, 다시 리뷰를 받는다.
버그가 0이 될 때까지 이 과정을 반복한다.

이건 에이전트 성능과 무관한 **구조적 차이**다.
에이전트가 뛰어나면 1회 만에 끝나고, 부족하면 여러 번 돈다.
어느 쪽이든 결국 수렴한다.

## 5. 조합별 상세 분석

### A. Agent Team — 가장 빠른 프로토타입

```
개발자 2명(병렬) → 코드 합치기 → 리뷰어 + 테스터(병렬)
```

- 평균 ~8분으로 가장 빠름
- 프로토타입이 급할 때 적합
- 단, Critical 버그가 항상 남음

### B. Agent Team + Worktree — 안전한 병렬 개발

```
개발자 2명(각자 격리된 공간) → 코드 합치기 → 리뷰어 + 테스터
```

- 개발자끼리 파일 충돌이 없음 (물리적 격리)
- 하지만 단일 기능 개발에서는 A와 차이 없음
- 여러 독립 기능을 동시에 만들 때만 가치 있음

### C. Ralph Loop + Team — 시간 대비 최고 품질 ⭐

```
개발자 → 리뷰어 + 테스터 → 버그 있으면 수정 → 재리뷰 → 버그 0까지 반복
```

- 평균 ~10분
- **8회 실험 모두 Critical 0 달성**
- 설정도 간단, 시간도 합리적
- 프로덕션 코드에 가장 적합

### D. Ralph Loop Parallel — 최고 품질, 최고 비용

```
개발자 → 리뷰어 + 테스터(병렬) → 수정 → 반복
```

- 평균 ~14분으로 가장 느림
- 테스트 수 최다 (평균 112개), 커버리지 100%
- 품질에 절대 타협 없을 때 사용

## 6. 시간 vs 품질 트레이드오프

```
 빠름                                          느림
  │                                            │
  A (~8분)      B (~8분)      C (~10분)      D (~14분)
  │                                            │
버그 있음                                     버그 없음
```

시간을 더 쓰면 품질이 올라가는 단순한 관계.
하지만 C(Ralph Loop + Team)가 **2분만 더 쓰고 버그 0을 달성**하므로 가성비가 가장 좋다.

## 7. 실험에서 발견한 3가지 인사이트

### 1. AI는 경험에서 배우지 않는다. 프롬프트에 적어야 배운다.

1차 실험에서 반복된 버그(무한 루프, alert 블로킹)를 프롬프트에 명시하자 2차부터 해당 버그가 사라졌다.

하지만 프롬프트에 없는 새로운 유형의 버그는 계속 나타났다.
반복 루프는 이런 **"예측 못한 버그"를 잡아주는 안전망**이다.

### 2. AI는 비결정적이다. 같은 조건에서도 결과가 다르다.

같은 조합이 어떤 날은 1회 만에 끝나고, 어떤 날은 2회가 필요했다.
시간도 ±2~3분 편차가 있었다.

하지만 **"버그 0 달성 여부"는 조합 구조에 의해 결정**되었다.
반복 루프는 이 변동을 자동으로 흡수한다. (부족하면 더 돌면 됨)

### 3. 역할 분리 + 스킬 규칙이 핵심이다.

"코드 잘 짜줘"는 안 통한다.
개발자, 리뷰어, 테스터로 역할을 나누고 각각에게 구체적인 **스킬 규칙**(총 19개)을 부여해야 한다.

```
개발자 → 7개 스킬: 컴포넌트 패턴, 스타일, 네이밍, 폴더 구조, 가독성, 접근성, 디자인
리뷰어 → 6개 스킬: 버그 탐지, 보안, 성능, 유지보수성, 가독성, 접근성
테스터 → 6개 스킬: 단위 테스트, 컴포넌트 테스트, API 모킹, E2E, 시각 회귀, TDD
```

## 8. 상황별 추천

| 상황 | 추천 조합 | 이유 |
|------|----------|------|
| 빠른 프로토타입 | A. Agent Team | ~8분, 최소 시간 |
| 병렬 독립 기능 | B. Team + Worktree | 파일 충돌 방지 |
| **프로덕션 코드** | **C. Ralph Loop + Team** ⭐ | **~10분, 버그 0, 간단** |
| 최고 품질 필요 | D. Ralph Loop Parallel | ~14분, 테스트 최다 |

**기본값: C. Ralph Loop + Team.**
에이전트가 좋으면 1회 만에 끝나고, 나쁘면 자동으로 여러 번 돈다.
어느 쪽이든 손해가 없는 구조다.

## 10. 환경 세팅 상세

### 도구

| 항목 | 값 |
|------|---|
| CLI | Claude Code v2.1.63 |
| 모델 | Claude Opus 4.6 (팀 리드) / Claude Sonnet 4.6 (개발·리뷰) |
| OS | macOS (Darwin 25.3.0) |
| 프레임워크 | Next.js 16.1.6 (App Router) + TypeScript 5 + TailwindCSS |
| 테스트 | Vitest + React Testing Library + jsdom |

### 에이전트 구성

Claude Code의 커스텀 에이전트 기능(`.claude/agents/`)을 활용해 3개 역할을 정의했다.

**개발자 (frontend.md)**
- 모델: Sonnet
- 스킬 7개: 컴포넌트 패턴, TailwindCSS, 네이밍 규칙, App Router 구조, 가독성/예측성/응집도/결합도, 접근성, 디자인
- 구현 순서: 스펙 → 타입 → 컴포넌트 → 스타일 → 접근성 → 빌드

**리뷰어 (frontend-reviewer.md)**
- 모델: Sonnet
- 스킬 6개: 버그 탐지, 보안, 성능, 유지보수성, 프론트엔드 4원칙, 접근성
- 신뢰도 80점 이상만 보고, Critical(90+) / Important(80-89) 분류

**테스터 (frontend-test.md)**
- 모델: Sonnet
- 스킬 6개: 단위 테스트, 컴포넌트 테스트, API 모킹, E2E, 시각 회귀, TDD
- 테스트 순서: 스펙 분석 → 훅 테스트 → 컴포넌트 테스트 → 통합 테스트

### 스펙 (기획서)

PM 에이전트 + spec-review 스킬로 작성한 통합 기획서:
- PRD: CRUD + localStorage, 기술 스택, In/Out of Scope
- 유저 플로우: 5단계 Primary Flow, 에러 플로우, 엣지 케이스 14개
- 에러 시나리오: S-1 (QuotaExceededError), S-2 (localStorage 접근 불가), D-4 (JSON 파싱 실패)
- 접근성: WAI-ARIA Tabs 패턴, aria-live 알림, 키보드 내비게이션

### 프롬프트에 명시한 주의사항

1차 실험에서 반복 발생한 버그를 프롬프트에 직접 명시:
- `saveToStorage` catch에서 `setTodos` 호출 금지 (무한 루프)
- `alert()` 사용 금지 (블로킹 UI → `role="alert"` div 사용)
- mutation 내부에서 직접 저장 (useEffect 저장은 안티패턴)
- IME `e.nativeEvent.isComposing` 체크
- `!hydrated` 시에도 `id="todo-list-panel"` 유지
- `TodoCount` 항상 렌더링
- JSON.parse 실패(D-4)는 `console.warn`만

### 조합별 실행 흐름

**A. Agent Team (1회성)**
```
TeamCreate → dev-1(타입+훅) + dev-2(컴포넌트) 병렬
           → reviewer + tester 병렬 → TeamDelete
```

**B. Agent Team + Worktree (1회성)**
```
TeamCreate → dev-1(worktree) + dev-2(worktree) 격리 병렬
           → 코드 머지 → reviewer + tester 병렬 → TeamDelete
```

**C. Ralph Loop + Team (반복)**
```
TeamCreate → dev(전체 구현) → reviewer + tester 병렬
           → Critical > 0 → dev 수정 → reviewer 재검증 → 반복
           → Critical 0 → TeamDelete
```

**D. Ralph Loop Parallel (반복)**
```
Agent(dev) → Agent(reviewer) + Agent(tester) 병렬
           → Critical > 0 → Agent(dev-fix) → Agent(reviewer) 재검증
           → Critical 0 → 종료
```
