# DESIGN.md — AI가 읽는 디자인 시스템

> 최종 수정: 2026-04-09

## TL;DR

DESIGN.md는 Google Stitch가 도입한 개념으로, 디자인 시스템을 마크다운 한 장에 담아 AI 코딩 에이전트가 일관된 UI를 생성하도록 하는 파일이다. 색상, 타이포, 간격, 컴포넌트 스타일 등 9개 섹션으로 구성되며, 프로젝트 루트에 넣으면 끝. VoltAgent의 awesome-design-md 레포는 Stripe·Linear·Figma 등 58개 기업의 DESIGN.md를 수집해 3일 만에 4,385 스타, 현재 36.2k 스타를 기록했다. "AI한테 UI 만들어달라고 했더니 매번 다른 색상, 다른 간격, 다른 버튼이 나오는" 디자인 슬럽(Design Slop) 문제의 현실적 해법이다.

---

## Part 1: 문제 — 디자인 슬럽(Design Slop)

### AI 코딩의 고질적 문제

AI에게 "로그인 페이지 만들어줘"라고 하면 매번:
- **다른 색상** — 어제는 파란 버튼, 오늘은 초록 버튼
- **다른 간격** — 컴포넌트마다 padding이 제각각
- **다른 스타일** — rounded-md였다가 rounded-full이었다가

이것이 **디자인 슬럽(Design Slop)** — AI가 생성하는 UI의 비일관성 문제다.

### 왜 발생하는가

| 원인 | 설명 |
|------|------|
| 컨텍스트 부재 | AI는 매 프롬프트마다 새로 시작, 이전 결정을 기억 못함 |
| 암묵적 지식 | "우리 프로젝트는 이런 느낌이야"를 코드로 전달할 수 없음 |
| 토큰 분산 | Figma, CSS, JSON 등 디자인 정보가 여러 곳에 흩어져 있음 |
| LLM의 기본값 | 명시적 지시 없으면 학습 데이터의 "평균적" 스타일을 생성 |

### 기존 해결 시도들

| 방법 | 한계 |
|------|------|
| Figma 링크 공유 | AI가 Figma를 직접 읽지 못함 |
| CSS 변수/Design Tokens (JSON) | 구조화되어 있지만 "분위기"나 "가드레일"을 담지 못함 |
| 프롬프트에 매번 설명 | 반복 작업, 누락 불가피 |
| Tailwind Config | 값은 정의하지만 "언제 어떻게 쓸지"는 없음 |

## Part 2: 해법 — DESIGN.md

### 개념

> **README.md가 프로젝트 설명서라면, DESIGN.md는 프로젝트 디자인 설명서다.**

- Google Stitch가 도입한 포맷
- 순수 마크다운 — 특별한 도구 불필요
- LLM이 네이티브로 이해하는 형식
- 프로젝트 루트에 두면 AI 에이전트가 자동으로 참조

### AGENTS.md vs DESIGN.md

| 파일 | 대상 | 정의 내용 |
|------|------|----------|
| `AGENTS.md` | 코딩 에이전트 | 프로젝트를 **어떻게 빌드**하는가 |
| `DESIGN.md` | 디자인 에이전트 | 프로젝트가 **어떻게 보여야** 하는가 |
| `CLAUDE.md` | Claude Code | 코딩 규칙 + 워크플로우 지시 |

### 9개 표준 섹션

```
1. Visual Theme & Atmosphere   — 분위기, 밀도, 디자인 철학
2. Color Palette & Roles       — 시맨틱 색상명 + HEX + 기능적 역할
3. Typography Rules            — 폰트 패밀리, 전체 계층 표
4. Component Stylings          — 버튼/카드/입력/네비게이션 + 상태별 스타일
5. Layout Principles           — 간격 스케일, 그리드, 여백 철학
6. Depth & Elevation           — 그림자 시스템, 표면 계층
7. Do's and Don'ts             — 디자인 가드레일, 안티패턴
8. Responsive Behavior         — 중단점, 터치 타겟, 축소 전략
9. Agent Prompt Guide          — 색상 참조 방법, 즉시 사용 가능 프롬프트
```

### 실제 예시 (Stripe 스타일 발췌)

```markdown
## Color Palette
- **Primary**: `#635BFF` (Stripe Purple) — CTA 버튼, 핵심 액션
- **Background**: `#F6F9FC` — 밝고 깨끗한 표면
- **Text Primary**: `#0A2540` — 본문, 헤딩
- **Success**: `#00D924` — 결제 완료, 긍정 피드백

## Typography
- Font: Inter, system-ui
- Heading: weight 600, tracking -0.02em
- Body: weight 400, line-height 1.6

## Do's and Don'ts
✅ 버튼은 항상 충분한 패딩 (12px 16px 이상)
✅ 카드는 subtle shadow + 1px border
❌ 네온 컬러나 강한 그라디언트 사용 금지
❌ 둥근 모서리 8px 초과 금지
```

## Part 4: Google Stitch와의 연동

### Google Stitch란

Google Labs에서 만든 AI UI 디자인 도구. Gemini 기반으로 UI를 생성하며, DESIGN.md를 컨텍스트로 사용한다.

### URL → DESIGN.md 자동 추출

1. Stitch에 웹사이트 URL 입력
2. Stitch가 자동 분석:
   - Color palette (primary, secondary, accent, background, text)
   - Typography (font families, sizes, weights, line heights)
   - Spacing & layout (grid, padding, margin 패턴)
   - Component style (버튼, 카드, 보더, 그림자)
3. DESIGN.md로 내보내기

### 활용 시나리오

| 시나리오 | 방법 |
|---------|------|
| 경쟁사 디자인 참고 | URL 입력 → DESIGN.md 추출 → 수정 후 사용 |
| 기존 프로젝트 정리 | 내 사이트 URL → 현재 디자인을 문서화 |
| 새 프로젝트 시작 | awesome-design-md에서 참고 기업 선택 |
| 클라이언트 요청 | "이 사이트처럼" → URL에서 추출 → 커스텀 |

## Part 5: Claude Code에서 활용하기

### 6단계 워크플로우

```
1. DESIGN.md 준비     — Stitch 추출 / awesome-design-md 복사 / 직접 작성
2. 프로젝트 루트 배치  — /project-root/DESIGN.md
3. CLAUDE.md 설정     — "UI 생성 시 DESIGN.md 참조" 지시 추가
4. Claude Code 시작   — 세션에서 DESIGN.md를 컨텍스트로 인식
5. 컴포넌트 생성      — "DESIGN.md 따라서 버튼 만들어줘"
6. 검증              — 색상값, 폰트, 간격 일치 확인
```

### CLAUDE.md에 추가할 내용

```markdown
## Design System
- UI 컴포넌트 생성 시 반드시 DESIGN.md를 참조할 것
- DESIGN.md에 정의된 색상, 폰트, 간격만 사용
- 새로운 값을 임의로 만들지 말 것
- 컴포넌트 상태(hover, active, disabled)는 DESIGN.md 패턴을 따를 것
```

### 우리 하네스에서의 위치

```
Phase 0: 브레인스토밍
Phase 1: 기획 (PM)
Phase 2: 기획 검증
Phase 3: UI 설계 ← DESIGN.md 생성 시점
Phase 4: 구현 계획
Phase 5: Foundation ← DESIGN.md를 프로젝트에 배치, Tailwind Config 매핑
Phase 6: 병렬 개발 ← 에이전트들이 DESIGN.md 참조하며 구현
Phase 7: 검증 루프 ← web-design-guidelines로 DESIGN.md 준수 여부 검증
```

**핵심 포인트**: DESIGN.md(입력)와 web-design-guidelines(검증)가 짝을 이룬다.
- DESIGN.md = "이렇게 만들어라" (사전 지시)
- web-design-guidelines = "이렇게 만들었나?" (사후 검증)

## Part 6: 한계와 고려사항

### DESIGN.md의 한계

| 한계 | 설명 |
|------|------|
| 정적 문서 | 디자인 변경 시 수동 업데이트 필요 |
| 해석 차이 | AI마다 같은 문서를 다르게 해석할 수 있음 |
| 복잡한 인터랙션 | 애니메이션, 마이크로 인터랙션은 마크다운으로 표현하기 어려움 |
| Figma 동기화 없음 | 디자이너의 Figma와 자동 싱크되지 않음 |
| 컴포넌트 라이브러리 미포함 | 실제 코드가 아닌 "설명서"일 뿐 |

### awesome-design-md의 한계

- 공개 CSS에서 추출한 값이므로 내부 디자인 토큰과 다를 수 있음
- 기업의 디자인 시스템은 수시로 변경되므로 시점 차이 발생
- 자동차 브랜드 등 일부는 웹 UI보다 마케팅 사이트 기준

### DESIGN.md vs 기존 도구 비교

| 도구 | 형식 | AI 호환성 | 분위기 전달 | 가드레일 |
|------|------|----------|-----------|---------|
| Figma | 비주얼 | ❌ 직접 불가 | ✅ 우수 | ❌ |
| Design Tokens (JSON) | 구조화 데이터 | ⚠️ 파싱 필요 | ❌ | ❌ |
| Tailwind Config | JS/TS | ⚠️ 부분적 | ❌ | ❌ |
| Storybook | 코드 + UI | ⚠️ 복잡 | ✅ 우수 | ❌ |
| **DESIGN.md** | **마크다운** | **✅ 네이티브** | **✅ 가능** | **✅ 가능** |

## 핵심 인사이트

1. **DESIGN.md는 "AI 네이티브" 디자인 시스템이다.** Figma나 JSON 토큰이 인간/기계를 위한 포맷이라면, 마크다운은 LLM이 가장 잘 이해하는 포맷이다. 디자인 시스템의 대상이 "개발자"에서 "AI 에이전트"로 확장되면서, 포맷도 바뀌는 것이 자연스럽다.

2. **입력(DESIGN.md)과 검증(web-design-guidelines)은 쌍이다.** DESIGN.md만 있으면 "참고는 했는데 맞게 했는지 모름", 검증만 있으면 "기준이 없는데 뭘 검사하나". 우리 하네스에서 Phase 3(생성) → Phase 7(검증) 루프가 완성된다.

3. **36k 스타는 "디자인 슬럽"이 보편적 고통임을 증명한다.** 3일 만에 4k 스타는 이 문제가 얼마나 절실했는지를 보여준다. AI 코딩이 "동작하는 코드"에서 "일관된 UI"로 관심이 이동하고 있으며, DESIGN.md는 그 전환점의 첫 번째 실용적 답안이다.
