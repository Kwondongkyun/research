# Pretext — CSS 없이 텍스트 레이아웃을 500배 빠르게

> 최종 수정: 2026-04-06

## TL;DR

브라우저한테 "이 텍스트 높이 얼마야?" 묻는 순간 페이지 전체가 다시 그려진다. Pretext는 이 질문 자체를 없앤다. 텍스트를 미리 분석해서 크기를 저장해두고, 이후로는 저장된 숫자를 더하는 것만으로 레이아웃을 계산한다. 결과적으로 기존 대비 약 500배 빠르고, DOM이 없는 Canvas·SVG에서도 쓸 수 있다.

---

## Part 1: 왜 기존 방식이 느린가

### 브라우저가 텍스트 크기를 알려주는 방식

웹에서 텍스트 높이를 알려면 브라우저한테 직접 물어봐야 한다.

```js
element.getBoundingClientRect() // "이 텍스트 높이 얼마야?"
```

문제는 이 질문을 받는 순간 브라우저가 **페이지 전체 레이아웃을 다시 계산**한다는 것. 이걸 DOM reflow라고 한다.

```
텍스트 높이 요청 → 브라우저가 실제로 그려봄 → 크기 알려줌
                         ↑
                  이 순간 페이지 전체 재계산
```

텍스트가 10개면 10번, 1000개면 1000번. 가변 높이 리스트에서 스크롤할 때마다 성능이 떨어지는 이유가 이것이다.

### Canvas는 다르다

브라우저에서 텍스트 너비를 재는 방법이 두 가지다.

| 방법 | 코드 | 특징 |
|---|---|---|
| DOM | `element.getBoundingClientRect()` | 페이지 전체 재계산 발생 |
| Canvas | `ctx.measureText("hello").width` | 재계산 없이 바로 계산 |

Canvas의 `measureText()`는 실제로 화면에 그리지 않아도 텍스트 너비를 알 수 있다. Pretext는 이걸 활용한다.

## Part 2: Pretext는 어떻게 동작하는가

API는 딱 두 개다. `prepare()`와 `layout()`.

### prepare() — 딱 한 번만 실행

텍스트가 처음 등장할 때 한 번 실행한다.

```ts
const prepared = prepare("안녕하세요 Hello 🚀", "16px Inter")
```

내부에서 세 가지를 한다.

**1. 텍스트 쪼개기**

`Intl.Segmenter`로 텍스트를 의미 단위로 분리한다.

```
"안녕 Hello 🚀" → ["안", "녕", " ", "Hello", " ", "🚀"]
```

- 영어: 단어 단위 (`"Hello"` 통째로)
- 한중일: 글자 단위 (`"안"`, `"녕"` 각각)
- 이모지·공백: 별도 처리

**2. 각 조각의 너비를 Canvas로 측정 후 캐시**

```
"Hello" → 38.5px  (저장)
"안"    → 16px    (저장)
"녕"    → 16px    (저장)
```

같은 단어가 여러 번 나오면 캐시에서 꺼낸다. Canvas를 다시 쓰지 않는다.

**3. 브라우저·이모지 보정**

Chrome/Firefox는 이모지를 Canvas에서 DOM보다 크게 측정하는 버그가 있다. 이걸 자동 감지해서 보정한다. Safari vs Chromium 동작 차이도 내부적으로 처리한다.

---

### layout() — 리사이즈할 때마다 실행

```ts
const { height, lineCount } = layout(prepared, 300, 20)
// maxWidth=300px, lineHeight=20px
```

저장해둔 숫자들을 더해서 "몇 줄인지, 총 높이는 얼마인지" 계산한다.

```
줄 1: "안"(16) + "녕"(16) + " "(4) + "Hello"(38.5) = 74.5px → 300px 이내 → 계속
줄 2: ...
```

브라우저한테 아무것도 묻지 않는다. 순수 산수다.

---

### 핵심 패턴

```ts
// 텍스트가 바뀔 때만 → prepare 다시 실행
const prepared = useMemo(() => prepare(text, "16px Inter"), [text])

// 컨테이너 크기가 바뀔 때 → layout만 다시 실행 (빠름)
const { height } = layout(prepared, containerWidth, 20)
```

`prepare()`는 비싸지만 1회, `layout()`은 싸고 N회. 이게 핵심 설계다.

---

### 성능 수치

| | 기존 (DOM) | Pretext |
|---|---|---|
| 측정 방식 | 브라우저가 실제로 그려봄 | Canvas로 미리 재둠 |
| 레이아웃 계산 | 매번 브라우저에 물어봄 | 저장된 값으로 덧셈만 |
| 500개 텍스트 기준 | ~30ms+ per frame | prepare 19ms (1회) + layout 0.09ms |
| 속도 차이 | 기준 | **~500배 빠름** |

## Part 3: 무엇이 가능해졌나

기존에 CSS나 DOM으로 구현하기 어려웠던 것들이 열렸다.

### 가변 높이 가상화 (120fps)

`react-virtual` 같은 가상화 라이브러리는 각 아이템 높이를 미리 알아야 한다. 기존엔 렌더링 후 DOM에서 측정했는데, Pretext는 렌더링 전에 높이를 알 수 있다. 수십만 개 아이템을 120fps로 스크롤 가능.

### Shrink-wrap 채팅 말풍선

CSS `fit-content`는 마지막 줄이 짧으면 그 줄 길이로 말풍선이 좁아진다.

```
┌────────────┐        ┌────────────┐
│ 안녕하세요    │  CSS   │ 안녕하세요    │
│ ㅎ          │   →    └────────────┘        ← 어색
└────────────┘
```

Pretext는 "줄 수를 유지하면서 가장 좁은 너비"를 계산할 수 있다. `walkLineRanges()`가 이걸 한다.

### Canvas·SVG 텍스트 렌더링

DOM 없이도 텍스트 레이아웃을 계산하기 때문에 Canvas, SVG, WebGL 어디서든 멀티라인 텍스트를 정확히 배치할 수 있다.

## Part 4: 언제 쓰고, 언제 안 쓰나

### 쓸 때

- 가변 높이 아이템 수천~수만 개 가상화
- Canvas/SVG에 텍스트 직접 렌더링
- 채팅 UI에서 말풍선 shrink-wrap
- 텍스트 높이 계산이 실제 성능 병목인 경우

### 안 써도 될 때

- 일반적인 웹페이지, 대시보드
- 텍스트 아이템이 수십 개 수준
- CSS로 이미 잘 되는 레이아웃

### 주의사항

- `system-ui` 폰트는 macOS에서 정확도가 떨어진다. `Inter`, `Helvetica Neue` 등 명시적 폰트명을 써야 한다.
- `prepare()`를 같은 텍스트에 반복 호출하면 오히려 느려진다. `useMemo`로 캐시해야 한다.

## Part 5: 기존 vs Pretext 비교

### 코드 비교

**가변 높이 리스트 아이템**

```tsx
// 기존 방식 — 렌더링 후 DOM에서 측정 → reflow 발생
function ListItem({ text }: { text: string }) {
  const ref = useRef<HTMLDivElement>(null)
  const [height, setHeight] = useState(0)

  useEffect(() => {
    setHeight(ref.current?.getBoundingClientRect().height ?? 0)
  }, [text])

  return <div ref={ref}>{text}</div>
}
```

```tsx
// Pretext 방식 — 렌더링 전에 이미 높이를 앎 → reflow 없음
function ListItem({ text, containerWidth }: { text: string; containerWidth: number }) {
  const prepared = useMemo(() => prepare(text, "16px Inter"), [text])
  const { height } = layout(prepared, containerWidth, 24)

  return <div style={{ height }}>{text}</div>
}
```

**채팅 말풍선 너비**

```tsx
// 기존 방식 (CSS fit-content)
// 마지막 줄이 짧으면 말풍선이 어색하게 좁아짐. 해결 불가.
<div style={{ width: "fit-content", maxWidth: 300 }}>
  {message}
</div>
```

```tsx
// Pretext 방식 — 줄 수 유지하면서 가장 좁은 너비 탐색
const prepared = useMemo(() => prepareWithSegments(message, "16px Inter"), [message])

let tightWidth = 0
walkLineRanges(prepared, 300, line => {
  if (line.width > tightWidth) tightWidth = line.width
})

<div style={{ width: tightWidth }}>{message}</div>
```

| 리사이즈 시 | 500번 DOM 접근 | 저장값 덧셈만 |

## 핵심 인사이트

1. **DOM reflow가 느린 이유는 측정 자체가 아니라 측정 방식** — Canvas `measureText()`는 DOM reflow 없이 텍스트 너비를 잰다. 이 차이 하나가 500배 성능 차이를 만든다.

2. **prepare/layout 분리가 핵심 설계** — 비싼 작업(Canvas 측정)은 1회, 싼 작업(덧셈)은 N회. 리사이즈·스크롤처럼 반복 호출되는 상황에서 압도적으로 유리하다.

3. **DOM이 없어도 된다는 게 진짜 혁신** — Canvas, SVG, WebGL, 서버사이드 모두 지원 가능해진다. 텍스트 레이아웃이 브라우저 렌더링 파이프라인에서 분리됐다.
